## 1. 文档目的

本文件定义 PLC 智能分选系统的总体技术架构、各层职责、信息流、控制边界和扩展接口，作为后续 I/O 点表、PLC 程序结构、HMI 画面、仿真场景及测试设计的上层依据。

本阶段仅进行工程设计，不生成 PLC 程序代码。

## 2. 设计输入与约束

### 2.1 设计输入

- 《PLC 智能分选系统需求分析》PSS-URS-001。
- 《西门子 PLC 平台选型报告》PSS-SEL-001。
- 《项目整体技术路线》PSS-TR-001。
- 《项目当前状态记录》PSS-STATUS-001。

### 2.2 实施约束

- 无实体 PLC、传感器、执行机构和电气控制柜。
- 控制平台采用 Siemens S7-1200，资源暂按 CPU 1214C 级别规划。
- 使用 TIA Portal 完成 PLC 与 HMI 工程设计，以 S7-PLCSIM 验证控制逻辑。
- WinCC HMI 在软件组件和许可证允许时实施；若暂不可用，先完成画面和变量接口设计。
- 急停只验证 PLC 的功能响应，不构成真实机器安全功能。
- Python 与 AI 属于后续扩展，不得进入急停、安全联锁或基本闭环控制链。

## 3. 总体设计原则

1. **分层设计：** 设备信号、PLC 控制、HMI、仿真和数据分析职责分离。
2. **单一控制源：** 每个最终输出只允许一个设备控制模块统一生成。
3. **接口稳定：** 控制核心通过标准化信号接口工作，不依赖信号来自真实 I/O 还是仿真模型。
4. **状态驱动：** 自动流程使用明确状态和转换条件，避免分散顺序逻辑互相覆盖。
5. **故障可诊断：** 每个关键动作具有许可条件、反馈、超时、报警和恢复路径。
6. **安全隔离：** HMI、Python 和 AI 均不能绕过 PLC 联锁直接驱动设备。
7. **可测试性：** 正常和故障场景能够重复触发，并能观察预期结果。

### 4.1 分层关系

| 层级 | 主要组成 | 核心职责 | 不承担的职责 |
|---|---|---|---|
| 操作与展示层 | WinCC HMI | 操作请求、状态显示、参数、报警和统计 | 不直接控制最终输出 |
| PLC 控制层 | S7-1200 控制逻辑 | 模式、联锁、顺控、设备、报警和数据 | 不模拟机械运动规律 |
| 设备与仿真层 | PLCSIM、虚拟设备模型 | 产生输入、反馈和故障，响应 PLC 输出 | 不决定生产控制策略 |
| 数据扩展层 | Python、数据文件、通信接口 | 统计、可视化、异常与趋势分析 | 不参与急停和实时联锁 |

## 5. 工艺对象与主要信息流

初版工艺对象包括一条输送线、入料检测、检测工位、分选位置、一个分选执行器、A/B 两条出料路径及出料确认。

### 5.1 物料流

`入料位置 → 输送 → 检测工位 → 类别锁存 → 分选位置 → A料道或B料道 → 出料确认`

### 5.2 控制流

`操作请求 → 启动许可检查 → 自动状态机 → 设备动作请求 → 设备模块联锁 → 最终输出`

### 5.3 反馈流

`虚拟传感器/执行器反馈 → 标准化输入 → 设备状态 → 工艺状态 → HMI与报警`

### 5.4 数据流

`分类结果/周期/报警/运行时间 → PLC统计与事件区 → HMI显示 → CSV或后续Python接口`

## 6. PLC 控制层设计

### 6.1 控制层职责

PLC 是系统唯一控制决策中心，负责：

- 系统停止、手动、自动和故障状态管理。
- 启动许可、急停状态、设备联锁和复位条件判断。
- 自动工艺步骤和步骤转换。
- 输送带、检测工位和分选执行器的动作管理。
- 物料分类结果锁存及当前位置跟踪。
- 动作超时、信号矛盾和过程异常诊断。
- 生产计数、周期时间及 HMI/数据接口维护。

### 6.2 逻辑分区

| 逻辑区域 | 职责 | 主要输入 | 主要输出 |
|---|---|---|---|
| 模式与启停管理 | 停止/手动/自动互斥、启动和复位 | HMI/按钮请求、许可状态 | 当前模式、运行状态 |
| 安全与许可管理 | 汇总急停、关键故障和启动条件 | 急停、设备故障、反馈 | 总许可、停机请求、原因码 |
| 自动顺控 | 管理工艺步骤、转换和异常退出 | 模式、许可、设备完成状态 | 各设备自动动作请求 |
| 输送控制 | 输出仲裁、运行反馈和超时 | 自动/手动请求、许可、反馈 | 输送最终命令、状态、报警 |
| 检测与分类 | 检测触发、结果锁存和有效性检查 | 检测位、分类信号 | 类别代码、检测完成/异常 |
| 物料跟踪 | 管理物料编号、类别和当前位置 | 工位事件、类别结果 | 当前物料数据、到位事件 |
| 分选控制 | 执行器动作、互锁、反馈和超时 | 分类、到位、手动请求 | 执行器最终命令、状态、报警 |
| 报警与诊断 | 报警产生、保持、确认和复位 | 各模块故障与状态 | 报警码、等级、汇总状态 |
| 生产统计 | 产量、分类数、异常数、周期和运行时间 | 完成与报警事件 | HMI和数据接口统计值 |

### 6.3 输出仲裁原则

自动顺控和手动调试只产生“动作请求”，不直接写最终设备输出。设备控制区域统一完成模式选择、安全许可、互锁、反馈检查和最终输出生成，从结构上避免多点写入和控制冲突。

### 6.4 自动控制状态

总体状态建议划分为：`停止`、`就绪`、`自动运行`、`正常停止中`和`故障锁定`。自动运行内部再划分为入料、检测、跟踪输送、分选和完成确认等工艺步骤。详细转换条件在后续状态机设计中确定。

### 6.5 数据区域规划

| 数据区域 | 内容 | 访问原则 |
|---|---|---|
| I/O 接口区 | 标准化输入、最终输出及设备反馈 | 由 I/O/仿真接口和设备模块使用 |
| 参数区 | 时间、阈值、批次目标和合法范围 | HMI 可按权限修改，PLC 校验 |
| HMI 接口区 | 操作请求、模式、步骤、设备状态和提示 | HMI 不直接访问设备输出 |
| 报警区 | 报警位、编号、等级、确认及恢复状态 | PLC 生成，HMI 展示与请求确认 |
| 生产数据区 | 物料、计数、节拍和运行时间 | PLC 维护，HMI/Python读取 |
| 仿真接口区 | 仿真使能、虚拟输入和故障注入 | 仅调试环境可用 |

## 7. HMI 监控层设计

### 7.1 HMI 定位

HMI 是操作与信息展示终端，不承担工艺顺序、设备联锁和安全控制。所有控制操作均以请求形式传给 PLC，由 PLC 判断是否接受并返回执行状态或拒绝原因。

### 7.2 画面架构

| 画面 | 主要内容 | 主要操作 |
|---|---|---|
| 系统总览 | 工艺流程、物料位置、模式、步骤、设备、产量 | 启动、停止、复位、画面导航 |
| 手动调试 | 设备状态、I/O、联锁原因 | 输送点动、分选单次动作 |
| 参数设置 | 动作时间、超时、批次目标、上下限 | 修改与确认参数 |
| 报警诊断 | 当前/历史报警、等级、时间、处理提示 | 报警确认、复位请求 |
| 生产统计 | 总数、A/B 数、异常数、节拍、运行时间 | 清零请求、数据查看 |
| 仿真面板 | 虚拟传感器、物料类别、故障注入 | 仅调试权限下使用 |

### 7.3 操作与权限原则

- 普通操作员可启停、复位、查看状态和确认允许的报警。
- 调试人员可进入手动和仿真画面、调整受控参数。
- 模式切换必须满足 PLC 条件，HMI 不通过隐藏按钮绕过联锁。
- 关键参数设置上下限，修改结果由 PLC 校验并反馈。
- HMI 通信中断时 PLC 按预定策略保持安全、可诊断的状态。

### 7.4 HMI 与 PLC 接口原则

- 命令采用瞬时请求或握手机制，避免按钮持续写入导致重复执行。
- 状态、报警和统计由 PLC 单向发布给 HMI。
- HMI 显示 PLC 返回的真实状态，而不是以按钮颜色代替设备反馈。
- HMI 急停图形仅用于状态显示或仿真触发，不能代表真实急停器件。

## 8. 仿真环境设计

### 8.1 仿真组成

| 组件 | 作用 |
|---|---|
| TIA Portal | 统一组态 PLC、网络、程序结构和 HMI |
| S7-PLCSIM | 运行虚拟 S7-1200、下载工程、在线监视与信号测试 |
| PLC 仿真模型 | 根据输出生成传感器和执行器反馈，模拟物料运动 |
| WinCC Runtime/仿真 | 验证操作、动画、报警、参数和统计交互 |
| 测试场景表 | 定义正常、停止、急停、超时和信号异常场景 |

### 8.2 仿真与控制解耦

控制核心只读取标准化输入并写入标准化输出。仿真模式下，虚拟设备模型向标准化输入提供信号；未来有硬件时，现场 I/O 向同一接口提供信号。切换数据来源不应修改自动顺控和设备控制的核心设计。

### 8.3 虚拟设备模型

- PLC 发出输送命令后，经过设定延时生成运行反馈。
- 虚拟物料按时序触发入料、检测位、分选位和出料传感器。
- 类别生成器按预设序列产生 A/B 类，保证测试可重复。
- 分选命令触发执行器到位与返回反馈。
- 故障注入可阻断反馈、保持传感器或生成无效分类值。

### 8.4 仿真模式控制

- 仿真使能仅在调试环境中使用并明显显示。
- 退出仿真或 PLC 重启后默认进入停止状态。
- 仿真面板不能直接写最终输出，只能改变虚拟输入或故障条件。
- 所有 FAT 用例记录输入场景、预期响应、实际结果和证据。

### 8.5 仿真边界

软件仿真可以验证控制结构、顺序、互锁、报警和 HMI 交互，但不能完全验证真实电气回路、机械惯性、传感器安装误差、现场网络质量和功能安全性能。

## 9. Python 数据分析接口设计

### 9.1 接口目标

在不影响 PLC 实时控制的前提下，向 Python 提供结构稳定的运行和事件数据，用于项目展示、生产统计和后续异常分析。

### 9.2 建议数据模型

| 数据类别 | 建议字段 |
|---|---|
| 系统快照 | 时间戳、模式、运行状态、当前步骤、总许可 |
| 物料事件 | 物料编号、入料时间、类别、完成时间、周期、结果 |
| 报警事件 | 报警编号、等级、发生、确认、恢复时间、当前状态 |
| 生产统计 | 总数、A/B 数、异常数、平均节拍、运行/停机时间 |
| 参数记录 | 参数编号、单位、当前值、修改时间和修改结果 |

### 9.3 分阶段接口路线

1. 第一版由 HMI 或测试过程导出 CSV，Python 离线读取。
2. 第二版建立 PLC 数据交换区，选择与软件许可和 CPU 能力匹配的标准通信方式。
3. Python 只读取必要数据；如未来需要写入，只允许写分析请求或经 PLC 校验的非安全参数。
4. 通信中断不得影响 PLC 自动循环，PLC 控制优先级始终高于外部应用。

具体通信技术在确认 TIA Portal 版本、CPU 固件和许可证后单独选型，不在本架构阶段强行锁定。

## 10. AI 扩展接口设计

### 10.1 适用范围

- 报警频率和停机趋势分析。
- 节拍异常识别。
- 设备维护建议和故障原因辅助排序。
- 项目文档、测试数据和运行结果的辅助解释。

### 10.2 禁止范围

- 不直接控制输送带或分选执行器。
- 不生成或旁路急停、安全许可和关键联锁。
- 不在未经验证时自动修改 PLC 参数。
- 不以 AI 推断替代确定性的报警和故障处理逻辑。

### 10.3 架构边界

AI 位于 PLC 控制边界之外，通过 Python 读取历史或准实时数据并输出“建议”。建议结果单独显示，必须与 PLC 的确定性状态和报警区分；任何未来的控制影响均需人工确认、PLC 合法性校验和独立风险评估。

## 11. 关键接口与依赖

| 接口 | 方向 | 内容 | 设计责任 |
|---|---|---|---|
| HMI → PLC | 操作层到控制层 | 启停、模式、复位、参数和确认请求 | PLC 校验并执行 |
| PLC → HMI | 控制层到操作层 | 状态、步骤、联锁原因、报警和统计 | PLC 发布，HMI 显示 |
| 仿真 → PLC | 仿真层到控制层 | 虚拟传感器、反馈和故障 | 仿真模型生成 |
| PLC → 仿真 | 控制层到仿真层 | 输送和分选输出命令 | 设备模块生成 |
| PLC → Python | 控制层到数据层 | 快照、事件、报警和统计 | 数据接口区发布 |
| Python/AI → PLC | 默认禁止 | 后期仅考虑受限、可校验的非安全请求 | 另行评审 |

## 12. 架构验收准则

- 系统各层职责明确且没有重复控制责任。
- HMI、仿真、Python 和 AI 均不能绕过 PLC 控制与联锁。
- 自动、手动、停止、急停和故障状态的信息路径完整。
- 真实 I/O 与仿真信号可以通过标准接口替换而不改变控制核心。
- 后续 I/O 点表中的每个信号能够归属到一个设备和接口。
- 后续 PLC 程序中的每项功能能够归属到一个逻辑区域。
