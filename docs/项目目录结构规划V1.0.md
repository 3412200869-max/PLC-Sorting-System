## 1. 文档目的

本文件基于阶段2已完成的系统架构、I/O、状态机、报警、HMI和FAT设计，规划 PLC-Sorting-System 的长期Git目录结构，使项目能够同时管理工程文档、TIA Portal工程、WinCC资料、PLCSIM仿真、Python扩展、测试证据和发布版本。

本次只建立目录规划，不创建实际TIA Portal、WinCC、PLCSIM或Python工程文件，也不移动现有项目文件。

## 2. 目录设计原则

1. **文档与实现分离：** `docs/`保存工程设计与验收资料，`PLC/`、`HMI/`、`Simulation/`、`Python/`保存实施成果。
2. **单一主工程：** TIA Portal集成项目只保留一个受控主归档，避免PLC与HMI目录分别保存互相冲突的完整工程副本。
3. **二进制与可审查内容并存：** TIA归档包用于恢复工程，同时导出PLC源文件、标签表、接口和版本说明，便于Git审查。
4. **测试可追溯：** 仿真场景、FAT用例、执行证据和缺陷记录按稳定编号关联。
5. **数据与安全隔离：** 示例数据可提交；账号、许可证、密钥、个人路径和敏感数据不得进入仓库。
6. **逐步创建：** 目录在对应工作包开始时创建，不提前加入无内容占位文件。
7. **路径可移植：** 推荐使用英文ASCII目录名，中文说明写入README和文档，降低工具兼容风险。

### 3.1 最终顶层目录基线

V1.0正式确定以下目录为实施阶段的顶层结构：

| 顶层目录 | 必需内容 | 基线决定 |
|---|---|---|
| `docs/` | 系统设计、I/O设计、状态机、报警、HMI设计、FAT规范和项目日志 | 所有受控工程文档统一入口 |
| `PLC/` | TIA Portal主工程归档、PLC导出源、标签和编译记录 | 保存唯一受控TIA主工程归档 |
| `HMI/` | WinCC画面、标签、报警、素材和导出资料 | 不重复保存另一份TIA主工程 |
| `Simulation/` | PLCSIM场景、监控表、故障注入、测试数据和证据 | 与FAT用例编号对应 |
| `Python/` | 数据分析、接口、测试、示例数据和报告 | 与PLC控制隔离，默认只读 |

`media/`和`releases/`为推荐配套目录，用于展示材料和发布清单；它们不改变以上五个核心实施目录。

### 3.3 TIA与WinCC文件唯一性规则

- TIA Portal集成项目的完整归档只保存在`PLC/tia-portal/archives/`。
- 如果WinCC HMI与PLC位于同一TIA项目中，`HMI/wincc/`只保存画面规格、导出资料、标签、报警配置、素材和截图。
- 如果未来使用独立WinCC工程，必须在`HMI/wincc/README.md`中说明其与TIA主工程的版本关系，仍不得出现两个均被称为“主工程”的副本。
- PLCSIM场景、监控表和证据保存在`Simulation/`，不嵌入文档目录作为唯一副本。
- Python生成的数据和报告不得成为PLC运行所依赖的控制输入。

## 5. `docs/` 工程文档目录

### 5.1 `01-project-management/`

保存项目状态、路线图、日志、阶段总结、风险、开放项和仓库规划，是项目管理的唯一受控入口。阶段切换时优先更新此目录。

### 5.2 `02-system-design/`

保存需求分析、PLC平台选型、系统总体架构和整体技术路线。该目录回答“系统为什么这样设计”，是后续I/O和程序实现的上层依据。

### 5.3 `03-io-design/`

保存I/O点表、变量命名规范和TIA Portal标签表规划。I/Q绝对地址变化时只更新映射和点表，核心功能名称保持稳定。

### 5.4 `04-state-machine/`

保存九状态自动流程、状态转换、正常停止、急停、复位恢复和PLC程序结构设计。后续程序实现必须与状态编号和转换优先级一致。

### 5.5 `05-alarm-design/`

保存18项报警、等级、变量、触发条件、系统响应、确认/复位规则和报警响应矩阵。报警编号一旦用于HMI和测试，不随意重排。

### 5.6 `06-hmi-design/`

保存WinCC画面、导航、权限、PLC/HMI变量接口和用户操作规范。此目录是HMI工程组态的设计输入，不保存完整TIA二进制主工程。

### 5.7 `07-fat-test/`

保存FAT测试规范、需求追溯矩阵、测试执行记录、偏差清单和回归结果。规范与执行记录分开管理，避免测试结果覆盖原始验收标准。

### 5.8 `08-data-ai/`

保存Python数据接口、数据字典、AI扩展用途和安全边界。AI只用于分析与建议，不参与急停、联锁和基本控制。

### 5.9 `09-user-guide/`

保存环境安装、工程打开、PLCSIM操作、演示脚本和故障恢复说明，确保项目能够在其他计算机上复现和展示。

## 6. `PLC/` TIA Portal工程目录

### 6.1 目录结构

| 路径 | 作用 | 推荐内容 |
|---|---|---|
| `PLC/tia-portal/archives/` | 保存可恢复的TIA Portal主工程归档 | 阶段性`.zap*`归档或对应版本格式 |
| `exported-sources/` | 保存可文本审查的PLC导出源文件 | SCL源、块导出、结构说明 |
| `tag-tables/` | 保存PLC标签表导出 | DI、DO、常量、HMI接口CSV/XLSX导出 |
| `block-interfaces/` | 保存OB/FB/FC/DB接口快照 | 块清单、接口表、实例关系 |
| `compile-reports/` | 保存关键版本编译结果 | 编译摘要、警告处置记录 |
| `README.md` | 说明TIA版本、CPU、固件、打开和归档方法 | 每个工程基线必须更新 |

### 6.2 主工程规则

- TIA Portal中的PLC和HMI可以位于同一个集成项目；完整主归档只保存在 `PLC/tia-portal/archives/`。
- `HMI/`保存画面、标签、报警和素材的导出或说明，不另存一份失控的完整TIA主工程。
- 不将TIA自动备份、缓存、临时文件和本地用户设置纳入Git。
- 具体归档扩展名随TIA版本变化，目录规划使用`.zap*`表示，不在确认版本前锁定扩展名。

## 11. 文件命名规范

### 11.1 文档

- 稳定文档使用固定名称，内部通过文档版本和变更记录管理。
- 导出交付物可包含版本，例如 `io-list-v1.0.xlsx`。
- 日期统一使用 `YYYYMMDD`，避免`final`、`final2`、`最新版`等名称。

### 11.2 TIA归档

推荐格式：

```text
PLC-Sorting-System_TIA_<项目版本>_<YYYYMMDD>.<zap版本扩展名>
```

示例仅作格式说明：

```text
PLC-Sorting-System_TIA_v0.3.0_20260712.zapXX
```

其中`zapXX`在确认实际TIA Portal版本后替换，不能将示例扩展名直接用于正式工程。

### 11.3 测试证据

```text
<用例ID>_<场景>_<执行日期>_<序号>.<扩展名>
```

例如：`FAT-004_ClassA_20260712_01.png`。

## 12. Git版本与TIA工程对应规则

每个可演示或可回退基线应同时记录：

| 项目 | 示例/要求 |
|---|---|
| Git分支 | 功能分支合并到主分支后形成基线 |
| Git提交号 | 写入TIA目录README和发布清单 |
| Git标签 | 例如`v0.3.0`，具体版本策略另行确认 |
| TIA归档名 | 包含同一项目版本和日期 |
| CPU/固件 | 写入TIA目录README |
| 软件版本 | TIA、PLCSIM、WinCC均记录 |
| FAT状态 | Not Run、Partial、Pass及报告路径 |

禁止只提交TIA二进制归档而没有版本说明，也禁止更新导出源文件却不更新对应归档或差异说明。

## 13. `.gitignore` 规划

正式创建工程后应根据实际文件生成`.gitignore`，至少排除：

- TIA/WinCC自动备份、缓存、临时文件和编译缓存。
- Python虚拟环境、`__pycache__`、测试缓存和本地配置。
- Runtime日志、临时CSV、未脱敏生产数据和测试输出。
- 软件许可证、账号、访问令牌、私钥和机器专用配置。
- 编辑器和操作系统生成的无关文件。

不得在未知TIA版本时凭猜测大量忽略文件；应先创建最小测试工程，识别可恢复主文件和自动生成内容，再评审`.gitignore`。

## 14. 迁移与创建顺序

当前仓库已有文档采用较扁平结构。未来执行本规划时建议：

1. 在独立Git分支建立目标目录和各目录README。
2. 配置并验证`.gitignore`。
3. 移动现有文档并逐项修复README相对链接。
4. 检查Git差异，确保文件是移动而非内容意外丢失。
5. 确认软件环境和CPU后创建TIA主工程目录。
6. 创建HMI、Simulation和Python目录时只加入真实交付物。
7. 提交迁移并建立目录结构基线标签。

本次不执行以上迁移，不创建空工程或占位工程文件。
